数据库 | 数据库系统概论 | 概述

发表于 2021-06-21 分类于 学的理论儿 Valine：

本文是《数据库系统概论》的第6章的读书笔记。笔者认为，本章的重点可分为以下3个部分：

  1. 数据库是什么？与其他存储数据的方式相比，数据库的特点在于？（人工管理 -> 文件系统 -> 数据库系统的发展历程）
  2. 为什么需要DBMS？DBMS的发展经历了哪些历程？（数据库中数据的高共享性所带来的问题&数据模型）
  3. 结合DBMS以后，数据库系统的结构是怎样的？（三级模式-二级映像）

数据库&DBMS的概念

数据库定义

以下是本书中数据库的定义：

数据库是长期存储在计算机内、有组织的、可共享的大量数据的集合。数据库中的数据按一定的数据模型组织、描述和存储，具有较小的冗余度与较高的数据独立性和易扩展性，可为各种用户共享。
（from《数据库系统概论》p14）

数据库技术的诞生，实际上是人们对于数据存储、检索、维护等不断增长的应用需求的产物。受限于时代的物质条件及技术发展，数据管理技术经历了人工管理、文件系统、数据库系统三个阶段的发展，具体历程在此不再详述，如感兴趣可参考《数据库系统概论（第5版）》第7页。

与人工管理与文件系统相比，数据库系统具备以下几个特点：

  • 实现了数据整体（而不仅仅是数据的单位记录内部）的结构化。这种整体上的结构化使得数据不仅仅面向某个应用，而是面向整个系统，因此可以被多个用户、多个应用共享使用。这种数据共享可以减少数据冗余，同时避免数据之间的不相容性与不一致性。
  • 具备较高的数据独立性。这种数据独立性，指的是用户开发的应用程序不会过于依赖数据库的具体实现方式，从而提高了数据的可复用性。这具体体现为物理独立性（即用户的应用程序与数据库中数据的物理存储是相互独立的）与逻辑独立性（即用户的应用程序与数据库的逻辑结构是相互独立的）。这种数据独立性，源自数据库管理系统的 “三级模式-二级映像” 结构。

DBMS的定义

根据第一点，我们可知，数据库具备较高的数据共享性。这种共享性，在带来许多便利的同时，也会带来新的问题。例如，多个用户可以同时存取同一个数据库中的同一条数据，就可能导致不同用户之间相互干扰的情况。那么，如何安排不同用户的操作优先级？通过什么机制，可以事前预防、事后弥补因极端情况而导致数据被篡改的情景？这些都是我们需要解决的问题。

为了解决以上所提到的问题，我们引入了DBMS作为“控制中枢”，对数据库中的数据提供了行以下的控制操作：1. 安全性保护， 防止数据以不合法使用造成的泄密与破坏 2. 完整性检查， 将数据控制在有效的范围内，保证数据之间满足一定的关系 3. 并发控制， 控制协调多用户的并发操作 4. 数据恢复， 具备将数据库从错误状态回复到某一已知的正确状态的能力。因此，DBMS的功能可被概括为：

在数据库建立、运用和维护时对数据库进行统一控制，以保证数据的完整性和安全性，并在多用户使用数据库时进行并发控制，在发生故障后对于数据库进行恢复。
（from 《数据库系统概论》p14）

数据模型——现实世界的抽象

建设数据库系统的目的，是为了存储、处理现实世界中的任务。然而，目前的计算机并不能以自然语言作为直接的输入处理复杂的任务。在使用计算机处理此类任务时，要把现实世界中具体的人、物、活动，抽象转换成计算机能够处理的数据。而完成这项任务的工具，便是数据模型。因此，若要理解现有的数据库系统如何工作，就首先得对数据模型进行一定的了解。
在将现实世界的具体事物抽象为某一数据库系统所支持的数据模型时，我们需要经历现实世界 -> 信息世界 -> 机器世界这样的一个过程，分别用到了概念模型、逻辑模型与物理模型这三个工具。下文将主要介绍概念模型与逻辑模型两个部分。

概念模型（现实世界 -> 信息世界）

概念模型是现实世界到信息世界的第一层抽象，是数据库设计阶段使用的主要工具，最为常用的表示方法为实体-联系方法（Entity-Relationship approach），用E-R图描述现实世界的概念模型。

所涉及的概念如下：

  • 实体（entity）: 客观存在且可互相区别的事物。
  • 属性（attribute）: 实体所具有的某一特性。
  • 码（key）：唯一标识实体的属性集。
  • 实体型（entity type）：用实体名与其属性名集合表示某一实体，用于抽象、刻画同类实体。例如，学生（学号，姓名，性别，所在院系）就是一个实体型。
  • 实体集（entity set）：某一类型实体的集合。
  • 联系（relationship）：常指不同实体集之间的联系。

逻辑模型（信息世界 -> 机器世界）

数据模型由数据结构、数据操纵、数据的完整性约束条件三部分所组成，分别描述了系统的静态特性、动态特性与完整性约束调节。下面将从这3个方面，介绍数据库领域中主要的逻辑数据模型，即：层次模型、网状模型、关系模型。

层次模型

典型代表是IBM于1968年推出的的IMS（Information Management System），是第一个大型商用数据库管理系统。
数据结构
数据库中的基本层次联系的集合需要满足以下2个条件：1. 有且仅有一个结点没有双亲结点，称为根结点。 2. 根以外的其他结点有且仅有一个双亲结点。

数据操作&完整性约束

  • 数据操纵：查询、插入、删除、更新。
  • 插入时，若无双亲结点，则不能插入子女结点。
  • 删除时，若删除双亲结点，则删除相应的子女结点值。

特点
优点主要有：

  • 数据结构简单清晰。
  • 查询效率高，性能优于关系数据库，不低于网状数据库。
  • 提供了良好的完整性支持。

缺点主要有：

  • 只能表示一对多关系，无法处理更复杂的关系。
  • 由于上一个缺点，导致插入与删除有诸多限制，若一个子女结点有多个双亲结点，需要用冗余数据或虚拟结点救一下。

网状模型

典型代表是DBTG（Data Base Task Group）系统，又称CODASYL（Conference On Data System Language），于20世纪70年代被提出，并非实际的数据库系统软件，但其基本概念及方法被后人继承，推动了IDMS，DMS1100，IDS/2，IMAGE等产品的出现。

数据结构
数据库中的基本层次联系的集合需要满足以下2个条件：1. 允许一个以上的结点没有双亲结点。 2. 一个结点可以有多于一个的双亲结点。

数据操作&完整性约束

  • 完整性约束：支持记录码（即唯一标识记录的数据项的集合）的概念。
  • 保证一个练习中双亲记录和子女记录之间是一对多的关系。
  • 可以支持双亲结点与子女结点之间的部分约束条件。

特点
优点主要有：

  • 更直接的描述现实世界
  • 存取效率高

缺点主要有：

  • 结构复杂，不利于用户学习
  • 访问数据时必须选择适当的存取路径，加重了编写应用程序的负担。

关系模型

数据结构

  • 定义：详见第2章。
  • 关系必须满足特定的规范条件，即范式。

数据操作&完整性约束

  • 数据操纵：查询、插入、删除、更新。
  • 操作对象与结果都是关系，即若干元组的集合
  • 存取路径被隐藏，便于用户使用

特点

优点主要有：

  • 建立在严格的数学概念的基础上。
  • 数据结构简单清晰，易用
  • 存取路径对用户透明，具备更高的数据独立性与安全性

缺点主要有：

  • 查询效率相对较低，DBMS需要进行查询优化。

数据库系统的结构

从数据库系统内部的系统结构来看，数据库系统可分为三级模式、两层映像的模式结构。

三级模式结构

模式

也称逻辑模式，是数据库中全体数据的逻辑结构和特征的描述，是所有用户的公共数据视图。它不涉及底层的物理存储的细节，也不涉及上层的应用程序及具体的开发工具。

外模式

也称子模式或用户模式，是数据库用户能够看见和使用的局部数据的逻辑结构与特征的描述，是数据库用户的数据视图，是与某一应用有关的数据的逻辑表示。

内模式

也称存储模式，是数据物理存储结构及存储方式的描述，是数据在数据库内部的组织方式。

两层映像结构

外模式/模式映像

对于每一个外模式，系统都有一个外模式/模式映像，定义了该外模式与模式之间的对应关系。
当模式改变时，由数据库管理员对各个外模式/模式的映像作相应改变，可以保证外模式保持不变。而应用程序又恰恰是依赖于外模式的。因此，当模式改变时，应用程序不需要作出相应的修改，从而保证了数据与程序的逻辑独立性，简称数据的逻辑独立性。

模式/内模式映像

数据库只有一个模式，也只有一个内模式，所以模式/内模式映像是唯一的，定义了数据全局逻辑结构与存储结构之间的对英国。

当数据库的存储结构改变时，数据库管理员可以修改模式/内模式映像，使得模式不需要作相应的修改，从而保证了数据与程序的物理独立性，简称数据的物理独立性。
